# Phase 07-02: Multi-Club Database Schema and Onboarding

## Objective
Extend the existing club infrastructure with multi-club scoping, enabling clubs to own their tournaments, manage members, and operate independently through a dedicated admin dashboard.

## Pre-existing Infrastructure
Before this phase, the codebase already had:
- `clubs` table with full schema (name, slug, location, contact, facilities, etc.)
- `club_members` table (player-to-club membership with roles)
- `club_claim_requests` table (claim verification flow)
- `club_venues` table (many-to-many club-venue linking)
- Club onboarding page at `/clubs/onboard`
- Club management page at `/clubs/manage`
- API routes for clubs CRUD, managed clubs, claims, members, venues

## New Work (07-02)

### 1. Migration: Multi-Club Scoping
File: `supabase/migrations/20260311_multi_club_scoping.sql`
- Add `club_id` FK to `tournaments` table for club-scoped tournaments
- Add `club_id` FK to `club_members_roster` table for club-scoped rosters
- Add `zip` column to `clubs` table
- Add computed `is_active` column to `clubs` table
- RLS policies for club managers to create/update their club's tournaments
- RLS policies for club managers to manage their club's roster
- `club_dashboard_stats` view for quick dashboard stat queries

### 2. TypeScript Types
File: `src/lib/db/club-types.ts`
- `ClubDashboardStats` - dashboard statistics interface
- `ClubWithStats` - club with dashboard context
- `ClubMemberDetail` - member with player info
- `ClubTournament` - club-scoped tournament
- `ClubVenueLink` - club-venue relationship
- `ClubClaimDetail` - claim request with details
- `ClubAdminTab` - tab navigation type

### 3. Club Admin Dashboard
File: `src/app/clubs/[state]/[slug]/admin/page.tsx`
- Overview tab with stats cards and quick actions
- Members tab with pending approval and active member management
- Tournaments tab listing club tournaments
- Settings tab for editing club info
- Auth guard: only club manager or admin can access
- Responsive design matching existing UI patterns

## Technical Decisions
- Used existing `/api/clubs/managed` PATCH endpoint for settings saves
- Direct Supabase client calls for member management (approve/reject/role)
- Client-side auth check via `supabase.auth.getUser()` + player lookup
- Stats computed client-side from fetched data (avoids extra view query)
