1. **Add `hasMutualRequest` to `src/lib/db/partner-requests.ts`:**
   - New async function `hasMutualRequest(requesterId: string, targetId: string)`
   - Queries `partner_requests` where `requester_id = targetId` AND `target_id = requesterId` AND `status = 'pending'` AND `expires_at > now()`
   - Returns the first matching `PartnerRequest` row or `null`

2. **Update `src/app/api/partner/request/route.ts`:**
   - Import `hasMutualRequest` from partner-requests, `createMatch` from matches, `updateRequestStatus` from partner-requests
   - Make timeout configurable: `const timeoutMinutes = parseInt(process.env.PARTNER_REQUEST_TIMEOUT_MINUTES || "5", 10);` then use `timeoutMinutes * 60 * 1000` for expiry calculation
   - AFTER the `hasPendingRequest` duplicate check but BEFORE creating the new request:
     - Call `const mutualRequest = await hasMutualRequest(currentPlayer.id, target_id)`
     - If mutualRequest exists:
       - `await updateRequestStatus(mutualRequest.id, "accepted")`
       - `const match = await createMatch({ sport }, [currentPlayer.id, target_id])`
       - Return `NextResponse.json({ status: "auto_matched", match }, { status: 201 })`
     - If no mutual request: proceed with normal flow (unchanged)

3. **Fix `src/app/api/partner/expire/route.ts`:**
   - The board page calls this endpoint on load without auth headers, causing 401
   - Change auth logic: try to get auth user first. If authenticated (any role), allow the call. If not authenticated, fall back to cron secret check. This is safe because `expireStaleRequests()` only updates already-past-expiry requests.
   - Simplified logic:
     ```
     const supabase = await createClient();
     const { data: { user } } = await supabase.auth.getUser();

     if (!user) {
       // Not logged in -- check cron secret
       const cronSecret = request.headers.get("x-cron-secret");
       if (!cronSecret || cronSecret !== process.env.CRON_SECRET) {
         return 401;
       }
     }
     // Proceed with expireStaleRequests()
     ```

4. **Fix RLS policies in `src/lib/db/schema.sql`:**
   - First check `src/lib/supabase/server.ts` to see if it uses the service role key. If it does, RLS is bypassed on server and no changes needed.
   - If it does NOT use service role: the current partner_requests RLS policies compare `auth.uid()` directly to `requester_id`/`target_id`, but those columns contain `players.id` (not `auth.users.id`). Fix by replacing with subquery through players table:
     ```sql
     create policy "Players can view own requests" on partner_requests for select using (
       exists (select 1 from players where players.id in (requester_id, target_id) and players.user_id = auth.uid())
     );
     ```
   - Add a comment in schema.sql noting the fix with "-- FIXED: was comparing auth.uid() to player.id instead of user_id"
  </action>

1. **Create `src/lib/hooks/useSentRequests.ts`:**
   Follow the same pattern as `usePartnerRequests.ts` but for OUTGOING requests.

   ```typescript
   "use client";
   import { useEffect, useState, useCallback } from "react";
   import { createClient } from "@/lib/supabase/client";
   import type { PartnerRequest, SkillLevel } from "@/lib/types";
   import type { RealtimePostgresChangesPayload } from "@supabase/supabase-js";
   ```

   - Interface `SentRequestWithTarget` extends `PartnerRequest` with joined `target` field:
     ```
     target?: { id: string; display_name?: string; name?: string; avatar_url: string | null; skill_level: SkillLevel; sports: string[] }
     ```
   - Hook signature: `useSentRequests(playerId: string | null, onStatusChange?: (request: PartnerRequest, newStatus: string) => void)`
   - `fetchSentRequests`: query `partner_requests` where `requester_id = playerId`, `status = 'pending'`, `expires_at > now()`, with joined target data: `target:players!target_id(id, display_name, name, avatar_url, skill_level, sports)`, ordered by `created_at desc`
   - Realtime subscription on channel `sent-requests-${playerId}`:
     - Listen for INSERT on `partner_requests` filtered by `requester_id=eq.${playerId}` -> refetch
     - Listen for UPDATE on `partner_requests` filtered by `requester_id=eq.${playerId}`:
       - If `status !== 'pending'`, remove from sentRequests list
       - Call `onStatusChange?.(updated, updated.status)` so the board page can show a toast
   - Return `{ sentRequests, loading, refetch }`
   - Cleanup: remove channel on unmount

     ```json
     }
     ```
   - Note: Vercel cron sends requests with `x-vercel-cron-signature` header, not `x-cron-secret`. The expire route should also accept Vercel's built-in cron auth OR the env var secret. Update the expire route if needed to check `request.headers.get("authorization") === \`Bearer ${process.env.CRON_SECRET}\`` as a fallback.
